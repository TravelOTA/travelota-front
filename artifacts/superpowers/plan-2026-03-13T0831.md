# Project Preparation for Backend Integration

To ensure a fluid integration with the backend and eliminate hardcoded data, we need to establish a clear separation of concerns between our components/composables and the data fetching logic.

## Proposed Changes

### [API Infrastructure]

#### [MODIFY] [nuxt.config.ts](file:///Users/ferrera/Git/travelota-front/nuxt.config.ts)
- Add `runtimeConfig` to define `apiBaseUrl`.

#### [NEW] [useApi.ts](file:///Users/ferrera/Git/travelota-front/app/composables/useApi.ts)
- Create a standard wrapper around Nuxt's `useFetch` for centralized API calls, handling authentication, and base URLs.

### [Data Modeling]

#### [NEW] [schemas.ts](file:///Users/ferrera/Git/travelota-front/app/utils/schemas.ts)
- Define Zod schemas for `Agency`, `Auth`, `Itinerary`, etc., to ensure type safety and data validation.

### [Refactoring Composables]

#### [MODIFY] [useAgency.ts](file:///Users/ferrera/Git/travelota-front/app/composables/useAgency.ts)
- Replace hardcoded `globalAgency` with a fetcher that uses `useApi`.

#### [MODIFY] [useItinerary.ts](file:///Users/ferrera/Git/travelota-front/app/composables/useItinerary.ts)
- Refactor to fetch itinerary data from the backend.

### [Mocking Layer (Dev only)]

#### [NEW] [agency.get.ts](file:///Users/ferrera/Git/travelota-front/server/api/agency.get.ts)
- Create a server route in Nuxt to provide the current mock data via an API endpoint, allowing the frontend to use real API patterns during development.
